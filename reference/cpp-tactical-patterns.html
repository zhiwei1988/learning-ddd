DDD 战术模式的 C++ 写法

主流 DDD 资料的示例都是 Java／C#，这份表把模式翻译成现代 C++ 惯用法。 所有代码均在 clang 17 / -std=c++20 -Wall -Wextra 下编译通过并跑过断言。 定义对齐 DDD Reference。

决策表：先回答这个问题

「如果两个实例的所有属性都相同，它们是同一个东西吗？」——答案决定一切。 注意：这个问题只在某个限界上下文内有答案（钞票在钱包里是值对象，在央行追踪里是实体）。

维度值对象 Value Object实体 Entity
身份来自属性本身标识（id），与属性无关
「同一个」意味着属性全等 → 可互换id 相等，纵使属性不同
operator=== default（逐成员比）手写，只比 id。绝不 default
可变性不可变：无 setter，操作返回新对象可变：属性随生命周期变化
不可变怎么实现private 成员 + 只读接口。不要用 const 成员—
拷贝随便拷，天然线程安全谨慎；通常按引用/指针传递
生命周期无。用完即弃有。创建→变更→归档/销毁
成本几乎为零唯一性、并发、持久化都要管
默认选哪个默认选它只在真的需要追踪「这一个」时

Evans 的两句原话，记住就够用：

"The model must define what it means to be the same thing." （模型必须定义「是同一个东西」是什么意思。）

"When you care only about the attributes and logic of an element of the model, classify it as a value object. […] Treat the value object as immutable."

值对象：范式

class Money {
 public:
  Money(long long cents, std::string ccy)
      : cents_(cents), currency_(std::move(ccy)) {}

  // Identity IS the attributes → let the compiler compare them all.
  bool operator==(const Money&) const = default;

  // Side-effect-free function: returns a new value, mutates nothing.
  Money plus(const Money& other) const {
    if (currency_ != other.currency_)
      throw std::invalid_argument{"currency mismatch"};
    return Money{cents_ + other.cents_, currency_};
  }

  long long cents() const { return cents_; }

 private:
  long long   cents_;      // NOT const — see the trap below
  std::string currency_;
};
  • 没有 setter——这才是不可变的实现方式。
  • 所有操作 const 且返回新对象——Evans 的 side-effect-free functions。
  • 构造函数是校验的入口——非法的值对象根本造不出来（此处：币种不匹配直接抛）。
  • operator== 用 = default（C++20）；需要排序时用 auto operator<=>(const Money&) const = default;。

实体：范式

class Order {
 public:
  explicit Order(OrderId id) : id_(std::move(id)) {}

  // Identity ONLY. Attributes deliberately ignored.
  // NEVER `= default` here: it would compare lines_ too, and two
  // different orders with the same items would collapse into one.
  bool operator==(const Order& other) const { return id_ == other.id_; }

  void add_line(const Money& amount) { lines_.push_back(amount); }

 private:
  OrderId            id_;
  std::vector<Money> lines_;   // entities hold value objects
};
  • 只比 id。用 = default 就是把实体写成了值对象——头号事故。
  • 实体内部装满值对象是常态（vector<Money>），反过来极少。
  • Evans：Keep the class definition simple and focused on life cycle continuity and identity.

⚠ 陷阱：const 成员会废掉值语义

C++ 程序员看到「immutable」的第一反应，往往是把成员标 const。这个坑很安静： 能编译、能构造、能比较，直到你要赋值或把它塞进会重新分配的容器。

class BadMoney {
  const long long   cents_;      // "immutable", right?
  const std::string currency_;
};
BadMoney a{100, "CNY"}, b{200, "CNY"};
a = b;   // 💥

真实报错（clang 17）：

error: object of type 'BadMoney' cannot be assigned because
       its copy assignment operator is implicitly deleted
note: copy assignment operator of 'BadMoney' is implicitly deleted
      because field 'cents_' is of const-qualified type 'const long long'

可用下面这组 static_assert 在自己的代码库里体检值对象：

static_assert(std::is_copy_assignable_v<Money>);      // ✓
static_assert(std::is_move_assignable_v<Money>);      // ✓
static_assert(!std::is_copy_assignable_v<BadMoney>);  // const 成员的代价
static_assert(!std::is_move_assignable_v<BadMoney>);

结论：不可变靠接口（无 setter），不靠 const 成员。值对象必须保持可赋值、可移动—— 它的全部价值就在于能被自由拷来拷去。

顺带治好的病：基本类型偏执

值对象最实际的日常收益，是让编译器替你挡住语义事故：

常见判断速查

概念通常是但要注意
金额 / 币种值对象几乎没有例外
