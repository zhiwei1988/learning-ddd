DDD 全景地图

整个课程只有这一张地图。每课学的东西都能在这里找到位置。定义对齐 Eric Evans 官方的 DDD Reference（工作区内附 CC BY 4.0 副本，59 页，全部模式的权威定义）。

一句话总纲：DDD 的赌注是——复杂软件的成败取决于领域模型的质量， 而不是技术选型。它给你两个工具箱：战略设计决定「把系统切成几块、每块多重要」， 战术设计决定「每一块里面的代码长什么样」。先战略，后战术。

战略设计 Strategic Design

在系统与团队的尺度上工作。
子域在问题空间（业务本来的样子，你发现它）；
限界上下文在解空间（软件怎么切，你设计它）。

统一语言 Ubiquitous Language
团队与领域专家共用的一套精确词汇，直接出现在代码里。→ 第 1 课
限界上下文 Bounded Context
一个模型和它的语言保持一致适用的边界；边界外同一个词可以是另一个意思。→ 第 2 课
子域 Subdomain：Core / Supporting / Generic
业务本身的分块（问题空间）。核心子域是你赢得竞争的地方，设计精力优先砸在这里；通用子域买、支撑子域简单做。→ 第 3 课
上下文映射 Context Mapping
刻画限界上下文之间的协作关系（合作、防腐层、跟随者……）。

战术设计 Tactical Design

在单个限界上下文内部工作。
C++ 落地写法见 战术模式速查表。

实体 Entity
由标识（而非属性值）定义的对象：同一张订单改了地址还是那张订单。→ 第 4 课
值对象 Value Object
由属性值定义、无标识、不可变的对象：金额、地址、日期区间。→ 第 4 课 · C++ 写法速查
聚合 Aggregate
一簇实体/值对象组成的一致性单元，只能通过聚合根访问和修改。
仓储 Repository
让聚合的存取看起来像操作一个内存集合，隔离持久化细节。一个聚合根配一个仓储；接口在领域层，实现在基础设施层。→ 第 7 课
领域事件 Domain Event
领域中已经发生、业务方关心的事实：OrderConfirmed。过去式、不可变（值对象），携 id 不携引用，由根产生，跨边界靠它异步协作。→ 第 6 课
领域服务 Domain Service
不自然属于任何实体/值对象的领域操作，无状态。

术语一旦在测验中被你证明掌握，就会收入工作区词汇表（GLOSSARY.md），此后所有课程沿用同一叫法。
