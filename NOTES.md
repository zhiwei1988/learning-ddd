  试一试三个计分区，「你需要哪些分数」说不清，成了他一直只回「已完成」的真正原因）。课末统一这么写：
  「做完告诉我你答错了哪几道、当时选的是哪个，全对就说『全对』」，并点明本课有几个计分区。
  错题比总分信息量大，也正好是我要的证据（哪个点没吃透）。第 6 课课末已按此改写，后续课沿用。

## C++ 教学策略（有效，继续用）
- 第 4 课验证了一个好路子：**先把 DDD 概念挂到用户已有的 C++ 肌肉记忆上**
  （「你早就在写值对象了，只是不知道它叫这名字」），再教「什么时候不该这么写」。
- **C++ 代码必须真编译**。验证脚本在 **`code/`**（见 `code/README.md` 的跑法），不要放 scratchpad——
  那是 session-specific 的，换会话就没了。第 4 课示例已在 clang 17 / `-std=c++20 -Wall -Wextra` 下
  编译 + 跑 8 条断言（含 const 成员删除拷贝赋值的 static_assert）。
  首次编译就抓到两个错（assert 宏吃不下花括号初始化列表、漏 include <type_traits>）——不编译就发出去会很难看。
- 贴**真实的编译器报错**比描述有效得多（第 4 课贴了 const 成员那个坑的原始 clang 报错）。
- 第 5 课又验证了同一个路子，而且更进一步：**让 C++ 的类型系统把 DDD 概念变成看得见的东西**
  ——「按值持有 = 聚合内，按 id 持有 = 聚合外」，读一眼成员声明就知道边界在哪。
  这是 Java 教材给不了的角度（那边一切皆引用，两种情况长得一样），要继续挖这条线。
- 好用的手法：**用一条绿色断言证明一件坏事**。第 5 课的 `leaked_lines()` 泄漏 getter，
  `assert(limit < o.total())` 通过——不变量被破坏而全程无人吭声。比讲道理有说服力。
- 编辑器（LSP）没有 compile_commands.json，会用旧标准去解析 `.cc`，于是对 `operator<=>`、
  `= default` 比较运算符等报一堆假错。**以 `c++ -std=c++20` 的真实结果为准**，别被诊断带跑。
