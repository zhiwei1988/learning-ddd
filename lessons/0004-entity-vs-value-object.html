实体 vs 值对象：什么叫「同一个东西」

战术设计第一课。好消息：这块是 C++ 的主场——你其实早就在写值对象了，只是不知道它叫这个名字。

你在地图上的位置：战略设计（语言／边界／定价）已收官。 从这一课起，你站在一个限界上下文内部，开始写代码。 随时打开 DDD 全景地图 定位。

先热身：回顾检查

两道回扣第 3 课的判断题。凭记忆作答。

一个问题，分出两类对象

Evans 把这件事压缩成了一句话，我认为它是整个战术设计里最重要的一句：

The model must define what it means to be the same thing. （模型必须定义：「是同一个东西」到底是什么意思。） — Eric Evans, DDD Reference，"Entities"（第 12 页）

这不是哲学，是每天都要做的决定。拿一个概念，问自己一个问题：

「如果两个实例的所有属性都相同，它们是同一个东西吗？」

  • 是 → 值对象。两张 100 元，金额币种都一样，那就是一回事，可以互换。
  • 否 → 实体。两个客户都叫张伟、都住同一小区，他们仍然是两个人。搞混了要出人命。

反过来还有一个同样重要的检验，专门用来识别实体：「它的属性全变了，它还是它吗？」 一张订单改了地址、加了商品、改了金额——它还是那张订单。这就是 Evans 说的 a thread of continuity and identity, going through a lifecycle, though their attributes may change （一条贯穿时间的连续性与标识之线，纵使属性会变）。

钞票：同一个概念，两个答案

「一张 100 元人民币」是实体还是值对象？

这个问题没有脱离上下文的答案——这正是第 2 课的回旋镖：

你的钱包上下文

关心的是：我有多少钱

  • 钞票 = 值对象
  • 两张 100 元完全可以互换
  • 没人在乎冠字号是多少
  • 「同一个东西」= 金额与币种相同

央行防伪追踪上下文

关心的是：这张纸去过哪

  • 钞票 = 实体
  • 冠字号 AB12345678 独一无二
  • 被追踪一生：印刷→流通→销毁
  • 「同一个东西」= 冠字号相同

所以「X 是实体还是值对象」这个问题，和第 3 课的「X 是不是核心子域」一样， 脱离上下文无解。你得先问：在这个上下文里，我们凭什么说两个东西是同一个？

为什么 Evans 让你默认选值对象

新手的本能是「万物皆实体，都给个 id」。Evans 明确反对，理由很实在：

Tracking the identity of entities is essential, but attaching identity to other objects can hurt system performance, add analytical work, and muddle the model by making all objects look the same. （追踪实体的标识是必要的，但给其他对象附加标识，会损害系统性能、增加分析负担， 并且让所有对象看起来都一个样，从而把模型搞浑浊。） — Eric Evans, DDD Reference，"Value Objects"（第 13 页）

标识是有成本的。一旦一个东西有了 id，你就得管它的生命周期、管它的唯一性、 管它在数据库里的行、管并发修改时谁赢。而值对象没有这些负担——它可以随便拷贝、随便丢弃、 天然线程安全，因为它不可变。所以 Evans 的指令是：

Therefore: When you care only about the attributes and logic of an element of the model, classify it as a value object. […] Treat the value object as immutable. Make all operations Side-effect-free Functions that don't depend on any mutable state. Don't give a value object any identity […] — Eric Evans, DDD Reference，"Value Objects"

你的主场：C++ 值语义

这里有个对你的红利。Java／C# 程序员学值对象很痛苦——他们的世界里一切皆引用， 「不可变」「按值相等」全得靠纪律和样板代码去模拟（写 equals、hashCode、 到处 final、还得防着别人拿到引用改内部状态）。

而 C++ 的值语义就是母语。你写了这么多年 struct Point、 operator==、const 引用传参——那就是值对象，只是没人告诉你它有个 DDD 名字。 这一课对你来说不是学新东西，是给已有的肌肉记忆挂上正确的标签，然后知道什么时候不该这么写。

注意右边那条注释：Order 绝不能用 = default 生成 operator==。 一旦 default，它会连 lines_ 一起比——两张内容相同的不同订单就会被判定为同一张。 这是把「实体」写成了「值对象」，是本课要防的头号事故。 左边恰恰相反：Money 就该 default，因为它的身份就是它的属性。 同一个 C++ 特性，两边的正确答案相反——分野不在语法，在你对「什么叫同一个东西」的回答。

C++ 专属陷阱：别用 const 成员来实现「不可变」

Evans 说值对象要 immutable。C++ 程序员的第一反应往往是把成员标成 const。 这是个坑，而且是个安静的坑——类照样能编译、能构造、能比较，直到某天你想赋值或把它塞进 std::vector 里排序：

class BadMoney {
  const long long   cents_;      // "immutable", right?
  const std::string currency_;
};

BadMoney a{100, "CNY"}, b{200, "CNY"};
a = b;   // 💥

真实的编译器报错（clang 17，-std=c++20）：

error: object of type 'BadMoney' cannot be assigned because
       its copy assignment operator is implicitly deleted
   16 |   a = b;
      |     ^
note: copy assignment operator of 'BadMoney' is implicitly deleted
      because field 'cents_' is of const-qualified type 'const long long'
    9 |   const long long   cents_;
      |                     ^

const 数据成员会让拷贝赋值和移动赋值被隐式删除，于是你的值对象没法排序、没法放进会重新分配的容器、 没法被赋值——而值对象的全部价值恰恰在于能被自由地拷来拷去。你为了「不可变」， 亲手废掉了值语义。

正解：不可变靠接口，不靠 const 成员。成员是普通的 private 数据； 不提供任何 setter；所有操作都是 const 成员函数并返回新对象。 这样它既是不可变的（外部无法改），又保有完整的值语义（可赋值、可移动、可排序）。
左边 Money 那段正是这么写的—— 本课所有 C++ 代码都在 clang 17 / C++20 下编译并跑过断言，包括这个陷阱的 8 条 static_assert。
