统一语言：让代码说业务的话

DDD 落地的第一步不是画架构图，而是造一门语言——一门业务方和你的代码都说的语言。

你在地图上的位置：DDD 分两个工具箱——战略设计（切分系统）和战术设计（写好每一块的代码）。 统一语言是战略工具箱的地基，其他一切都建在它上面。 随时打开 DDD 全景地图 定位。

问题：每个项目里都住着两门语言

业务方说：「已支付但还没发货的订单，客户可以取消，钱退回去。」 代码却说：if (r.st == 2 && r.ship_flag == 0) r.st = 4;

这中间存在一个隐形的翻译过程：需求 → 开发者脑内翻译 → 技术实现。每一次翻译都有损耗， 而损耗会积累成 bug、误解和「这段代码到底在干嘛」。DDD 对此的判断很激进： 翻译本身就是错误的根源，所以要消灭翻译——让所有人、包括代码，说同一门语言。

A language structured around the domain model and used by all team members within a bounded context to connect all the activities of the team with the software. （一门围绕领域模型构建的语言，团队所有成员在一个限界上下文内用它把团队的一切活动与软件连接起来。） — Eric Evans, DDD Reference, "Ubiquitous Language"

关键词是 ubiquitous（无处不在）：这门语言不只活在需求文档里， 它必须出现在类名、方法名、变量名、测试名里。业务方在会议室说的那句话， 你应该能在代码里逐字找到。

同一条业务规则，两种写法

规则：「已支付且未发货的订单可以取消，取消后退款。」

注意右边发生了什么：cancel、paid、shipped、refund 全是业务方的原话；魔法数字消失了；而且业务规则（什么情况下不能取消）从「散落在调用方的 if」 变成了 Order 自己守卫的规矩。命名对齐了，职责往往会跟着对齐——这是统一语言送你的第一份礼物。

嗅探清单：你的代码在说哪门语言？

拿到任何一段业务代码，闻这三个味道：

  • 魔法值：状态用 2、4、"P" 表示——业务方在会议室里说不出「状态等于四」这种话。
  • 万能动词：满屏 update / process / handle / manage——业务动作（取消、退款、发货、作废）被压扁成了「改数据」。
  • 翻译对照表：需要一张 wiki 表格解释「代码里的 X 就是业务说的 Y」——对照表的存在本身就是语言分裂的自白。
