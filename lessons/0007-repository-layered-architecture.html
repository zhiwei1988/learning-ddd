仓储与分层架构：领域模型住哪、怎么存

你手上已经有了聚合、有了领域事件——但它们现在全活在内存里。这一课回答两个你上一课就追问过的问题： 聚合怎么存进去、取出来而不被数据库反噬；confirm → 保存 → 发布这套编排，到底住在哪一层。

你在地图上的位置：前几课都在单个聚合内部打转。这一课镜头拉远， 看整个系统怎么分层、领域模型怎么和数据库/消息总线打交道——这是你从「会写模式」迈向「会做架构」的第一步。 随时打开 DDD 全景地图 定位。

先热身：回顾检查

两道回扣前几课的题。凭记忆作答。

问题：领域模型正在被基础设施淹没

不分层的代码长什么样，你一定见过：Order 类里直接拼 SQL，业务规则散落在 UI 控件的回调和数据库脚本里。 Evans 一针见血：

When the domain-related code is diffused through such a large amount of other code, it becomes extremely difficult to see and to reason about. Superficial changes to the UI can actually change business logic. （当领域代码被稀释、散落在大量其他代码里，它就变得极难看清、极难推理。改一下界面，竟能改掉一条业务规则。） — Eric Evans, DDD Reference，"Layered Architecture"（第 10 页）

你上一课把聚合和事件都建好了，可它们一旦要落库、要真的把 OrderConfirmed 发出去， 就会有人忍不住往 Order 里塞一句 INSERT INTO ...。那一刻，你精心画的聚合边界就开始漏了。

解法：把系统切成层，让依赖只朝一个方向

Therefore: Isolate the expression of the domain model and the business logic, and eliminate any dependency on infrastructure, user interface, or even application logic that is not business logic. Partition a complex program into layers. Develop a design within each layer that is cohesive and that depends only on the layers below. （因此：把领域模型和业务逻辑隔离出来，剔除它对基础设施、界面、乃至任何非业务的应用逻辑的依赖。 把复杂程序切成若干层，每层内部内聚，且只依赖它下面的层。） — DDD Reference，"Layered Architecture"（第 10 页）

整节课其实只有一句话：依赖只朝里指。领域层坐在最里面，谁都不依赖；越往外，越依赖里面。

最外层

基础设施层 Infrastructure

数据库、消息总线、HTTP、第三方 SDK。它认识领域层（去实现领域定义的接口），领域层不认识它。

▲ 依赖朝里指（外层认识内层，内层不认识外层）

中间

应用层 Application

薄薄的编排：取出聚合 → 调用它 → 保存 → 发布事件。不含任何业务规则，只负责把用例串起来、管事务边界。

▲

最里层

领域层 Domain

实体、值对象、聚合、领域事件、业务规则。不依赖任何东西——没有数据库、没有框架。它是整个系统的心脏。

仓储：让存取「像操作一个内存集合」

那领域层不认识数据库，聚合怎么存？答案是仓储（Repository）。Evans 的处方：

For each type of aggregate that needs global access, create a service that can provide the illusion of an in-memory collection of all objects of that aggregate's root type. （为每一类需要全局访问的聚合，建一个服务，营造出「一个装着该聚合根全部实例的内存集合」的错觉。） — DDD Reference，"Repositories"（第 17 页）

调用方看到的是 repo.by_id(id)、repo.save(order)，像在操作一个集合， 而不是 SELECT ... JOIN ... WHERE。两条纪律，都直接接着上一课：

  • 一个聚合根，一个仓储。没有 OrderLineRepository——订单行在 Order 聚合内部， 只能经由根访问（第 5 课）。你存取的单位永远是整个聚合。
  • 仓储进出的是完整聚合，不是零散字段。Evans 警告过：绕过根、从聚合内部直接 new 出几个对象， 会blindside 掉聚合根，让它没法再守规则，实体于是沦为「data container」。这正是第 5 课那个泄漏 getter 的持久化版本。

C++：接口在领域层，实现在基础设施层

「依赖朝里指」翻译成 C++，就是一个再熟悉不过的东西——纯虚接口（port）声明在领域层，具体实现（adapter）放在基础设施层：

// ===== DOMAIN LAYER =====  (depends on NOTHING below)
class OrderRepository {                     // the PORT — an abstract interface
 public:
  virtual ~OrderRepository() = default;
  virtual std::optional<Order> by_id(OrderId) const = 0;   // collection illusion
  virtual void save(const Order&) = 0;
};

// ===== INFRASTRUCTURE LAYER =====  (depends on the domain, implements its port)
class SqlOrderRepository : public OrderRepository {   // the ADAPTER
  // ... this is the ONLY place that #includes <pqxx> / <sqlite3.h> ...
  std::optional<Order> by_id(OrderId) const override;  // translate rows -> Order
  void save(const Order&) override;                    // translate Order -> rows
};

在 C++ 里，「分层有没有塌」是一个看得见、grep 得到的事实——就看一行 #include。 领域层的头文件里一旦出现 #include <sqlite3.h>，依赖的箭头就指反了，分层当场塌掉。 Java 把一切都摊在 classpath 上，这个违规是隐形的；C++ 的物理依赖让它无所遁形—— 你为了编译速度斤斤计较的那些 include 卫生，恰好就是分层架构的体检项。

那 confirm → 保存 → 发布 住哪层？答案：应用层

这就是上一课留的问题。编排既不属于领域层（它不是业务规则），也不属于基础设施层（它不碰具体技术）。 它是应用层的活——一段薄薄的、只依赖抽象的协调代码：

// ===== APPLICATION LAYER =====  (orchestration only, NO business rules)
class ConfirmOrderService {
 public:
  ConfirmOrderService(OrderRepository& repo, EventPublisher& bus)  // depends on
      : repo_(repo), bus_(bus) {}                                   // ABSTRACTIONS

  void execute(OrderId id) {
    Order order = repo_.by_id(id).value();      // 1. load the whole aggregate
    order.confirm();                            // 2. run the domain rule (in Order)
    repo_.save(order);                          // 3. persist the new state...
    for (const auto& e : order.pull_events())   // 4. ...then announce the facts
      bus_.publish(e);
  }
 private:
  OrderRepository& repo_;   // never names SqlOrderRepository — only the port
  EventPublisher&  bus_;
};

注意 ConfirmOrderService 从头到尾没提过任何具体实现——它只认 OrderRepository& 和 EventPublisher& 两个抽象。 到底用 SQL 实现还是内存实现，是在组装根（composition root，通常就是 main）里才拍板、才接线的。 这带来一个立竿见影的红利：整个领域层和应用层，不需要数据库、不需要真的消息总线就能测——给它塞两个假的实现即可。

本课代码全部真编译过。code/lesson7_check.cc，clang 17 / -std=c++20 -Wall -Wextra 零告警。 它把三层堆在一个文件里（读的时候当三层看），断言了：应用服务在零数据库、零真实总线下跑通（依赖朝里的可测性红利）、 存进去能原样取回（仓储的集合错觉）、编排产出的事件到达了总线。 还有一条用绿色断言证明的坏事——见下。跑法见 code/README.md。

把上一课的伏笔收掉：为什么需要发件箱

上一课你问过「事件发出去、订阅方失败了怎么办」。现在你看到了更根本的裂缝： save() 和 publish() 是两次独立的写入（一次写库，一次写总线）。中间进程一崩，两边就对不上——而且没有任何一种先后顺序是安全的：

本课验证代码用一条绿色断言把左边这个坑演示了出来：事件发了，可 save 崩在提交前， 重新加载订单竟然是「未确认」——下游为一件并不持久的事实动了手，全程无人报错。

解法方向：发件箱（Outbox）。把事件和聚合状态写进同一个事务（事件先落进一张 outbox 表）， 事务提交后，再由一个后台进程可靠地把 outbox 里的事件投递出去。两次写变成一次写 + 一次可靠投递，裂缝就消失了。 这属于下一步的实现细节，这里先埋这个名字——你已经能看懂它要解决的到底是什么问题了。

主推读物

本课首选延伸阅读：Alistair Cockburn 的 Hexagonal Architecture（Ports & Adapters）—— 它是「依赖朝里指」这个思想最经典、最一般化的表述：领域在中心，外面全是可插拔的适配器（数据库、UI、消息总线都是）， 通过端口（就是你这一课写的纯虚接口）连接。你这一课的 OrderRepository 是端口，SqlOrderRepository 是适配器， main 是组装根——读完这篇，你会发现自己已经在用六边形架构了。 一手定义见 DDD Reference 第 10 页（分层）与第 17 页（仓储），本课引文都出自这两页。

有不清楚的？问我。这一课常见的卡点： 「应用层和领域层到底怎么分？」（——一句话：有 if/规则的算领域，只有『先做A再做B』的算应用。折扣怎么算是领域的，「取出来→算一下→存回去」是应用的） 「仓储要不要支持复杂查询、分页、报表？」（——聚合的存取用仓储；纯读的报表/列表另说，别硬塞进领域模型，这条线以后细讲） 「接口都放领域层，具体实现放基础设施，不就是依赖倒置吗？」（——正是。你已经在用 SOLID 的 D 和六边形架构了）
