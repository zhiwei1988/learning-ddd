上下文映射与防腐层：上下文之间怎么协作、怎么翻译

前四课你在一个上下文里把战术模式打磨得很利。可真实系统从来不止一个上下文——它们必须协作。 这一课回答你在第 2 课就问过、我一直欠着的那个问题： 两个上下文都要「客户」，字段是不是要抄两遍？

你在地图上的位置：第 4–7 课都在单个聚合/单个上下文内部。这一课镜头拉到最远—— 看多个上下文之间怎么连线。这是战略设计的收尾，也正是你 mission 里「主导一次领域建模」 最要紧的一块：划完上下文之后，怎么治理它们的关系。随时打开 DDD 全景地图 定位。

先热身：回顾检查

两道回扣第 7 课那条最细的线（领域/应用/基础设施）。凭记忆作答。

那个欠了六课的问题

你的生鲜电商里，「客户（Customer）」这个词，销售上下文和物流上下文都要用。 第 2 课你追问过：那客户的字段，是不是两边各抄一份？听起来很浪费，直觉是「抽个共享的 Customer 类大家复用」。 这个直觉，恰恰是 DDD 要拦住你的地方。

销售上下文 Sales

这里的「客户」= 会买东西的人

  • Customer：信用额度、会员等级、历史订单、优惠券
  • 关心的是「他能不能下这单、给不给折扣」

物流上下文 Shipping

这里的「客户」= 要收货的人

  • Consignee（收件人）：收货地址、门牌、联系电话、配送时间窗
  • 关心的是「货送到哪、几点送、谁签收」——不关心信用额度

看出来了吗：它们根本不是同一个东西。销售的 Customer 装着信用额度和优惠券， 物流的「收件人」装着地址和时间窗。硬抽一个「全字段大 Customer」给两边共用，结果是两边都被塞进一堆自己不关心的字段， 而且销售改一下会员逻辑，可能编译不过物流——这正是第 1 课统一语言的铁律： 同一个词，在不同上下文里可以是不同的模型。所以答案是：不抄字段——各自建各自的模型， 两边只在需要打交道的那一刻，做一次翻译。

上下文映射：先把「连线」如实画出来

可它们终究要打交道：物流要拿到收货人信息，而这信息的源头在别处。两个上下文之间这条连线， 就是 Evans 说的上下文映射（Context Map）要处理的东西。

When connections must be made between different contexts, they tend to bleed into each other. ... Identify each model in play on the project and define its bounded context. ... Describe the points of contact between the models, outlining explicit translation for any communication... （当不同上下文之间必须建立连接时，它们往往会互相渗透、彼此污染。……识别项目里每一个在用的模型、定出它的限界上下文…… 描述模型之间的接触点，为任何通信勾勒出显式的翻译。） — Eric Evans, DDD Reference，"Context Map"（第 29 页）

两个关键词：bleed into each other（互相污染）是病，explicit translation（显式翻译）是药。 上下文映射第一步不是设计、是如实描述现状：项目里有哪些上下文、哪两个之间有连线、每条连线是什么关系。 DDD 给了一套现成的关系模式当标签（合作、共享内核、客户/供应商、遵奉者、防腐层……全表见 关系模式速查表）。这一课我们只深挖其中最能保护你的一个。

先认一条主轴：上游 / 下游

大多数连线上，两个上下文地位不对等：上游（Upstream）变了，下游（Downstream）得跟着改；下游怎么折腾，上游不受影响。 一句话——「谁能不管谁的死活」。你的物流上下文要依赖一个老旧的 CRM 系统拿收件人资料：CRM 是上游，物流是下游。 麻烦在于，这个 CRM 的模型很烂——字段是拼在一起的字符串、用空串 "" 表示「没有」、还有个没人写文档的 status_flag = 9。身为下游，你怎么办？

解法：防腐层（Anticorruption Layer）

一个偷懒的选择是遵奉者（Conformist）——干脆盲从上游的模型，CRM 那套烂结构直接在物流代码里传来传去。 省事，但你精心设计的物流模型会被那些空串和神秘 flag 一点点同化。Evans 给下游的另一条路，是防腐层：

As a downstream client, create an isolating layer to provide your system with functionality of the upstream system in terms of your own domain model. This layer talks to the other system through its existing interface... Internally, the layer translates ... between the two models. （身为下游客户，建一层隔离层，把上游系统的功能以「你自己的领域模型」的措辞提供出来。 这层通过上游现有的接口与之通信……在内部，按需在两个模型之间翻译。） — DDD Reference，"Anticorruption Layer"（第 34 页）

物流上下文（你的模型）

干净：Consignee 一构造就合法、「未知」= optional 的缺席，没有神秘 flag

C++：防腐层就是你上一课写过的端口/适配器

这里有个让你会心一笑的地方：防腐层，你上一课已经会写了。第 7 课的 OrderRepository—— 端口在领域层、SqlOrderRepository 适配器在基础设施层，翻译「数据库行 ↔ 领域对象」。 把「数据库」换成「另一个上下文的模型」，一模一样的形状，就是防腐层：

// ===== UPSTREAM: the legacy CRM's model — someone else's ugly world =====
struct CrmContactRecord {
  std::string full_nm;      // combined name, may be padded with spaces
  std::string ctry_cd;      // "" is a SENTINEL meaning "unknown"
  int         status_flag;  // 9 = undocumented legacy junk
};

// ===== YOUR DOMAIN: Shipping's own clean model (knows nothing of the CRM) =====
class Consignee {                         // valid-by-construction
 public:
  Consignee(std::string name, std::optional<CountryCode> country);  // throws if name empty
  const std::optional<CountryCode>& country() const;  // "unknown" = honest absence
};

class ConsigneeDirectory {                // the PORT — phrased in YOUR terms
 public:
  virtual std::optional<Consignee> find(int id) const = 0;
};

// ===== THE ANTICORRUPTION LAYER — the ONLY place that names CrmContactRecord =====
class CrmConsigneeDirectory : public ConsigneeDirectory {   // the ADAPTER
  std::optional<Consignee> find(int id) const override {
    auto rec = crm_.lookup(id);           // talk to the foreign system as-is
    if (!rec) return std::nullopt;
    return translate(*rec);               // <-- anticorruption happens HERE
  }
  static Consignee translate(const CrmContactRecord& r) {
    std::optional<CountryCode> country;
    if (!r.ctry_cd.empty()) country = CountryCode{r.ctry_cd};  // ""  -> nullopt
    // r.status_flag is simply NOT part of our model -> it can never leak in.
    return Consignee{trim(r.full_nm), country};                // trim the padding
  }
};

看这个对照就懂了防腐层在防什么：CRM 用空串 "" 表示「国家未知」，用没人懂的 status_flag = 9 表示某种遗留状态。翻译层是唯一认识 CrmContactRecord 的地方—— 过了这道墙，你的物流代码只见得到干净的 Consignee：「未知」是老老实实的 std::optional 缺席， 而那个神秘 flag 压根不在你的模型里，想泄漏都无处可泄。这就是 Repository 的 SqlOrderRepository（挡数据库）升级成挡「另一个上下文」——同一把尺子，量到了战略层。

本课代码全部真编译过。code/lesson8_check.cc，clang 17 / -std=c++20 -Wall -Wextra 零告警。 它断言了：翻译层把带空格的名字 trim 干净、把哨兵 "" 变成 nullopt、上游查不到时返回 honest 的 nullopt。 还有一条用绿色断言证明的坏事：如果走遵奉者的路直接传 CrmContactRecord， 那个 "" 和 status_flag=9 就在原始记录里明晃晃地存在（assert(raw.ctry_cd == "") 通过）—— 而防腐层的产物里，它俩一个变成了诚实的缺席、一个根本不存在。同样的输入，污染被挡在了边界。跑法见 code/README.md。

把伏笔收干净：所以「客户」到底抄不抄

回到那个欠了六课的问题。答案现在完整了：

  • 不抄字段。销售的 Customer 和物流的 Consignee 是两个上下文各自的模型， 各装各关心的东西。硬抽一个共享大类，是把共享内核用错了地方—— 那玩意儿只该共享一小块、且要两队持续协商，代价很高。
  • 要打交道时，做显式翻译。物流需要收件人资料，就通过一个防腐层去上游取、翻译成自己的 Consignee。跨的是上下文边界，翻译层挡住上游模型的污染——这跟你 第 5 课「跨聚合只用 id 引用」是同一种智慧在不同尺度上的体现： 边界之外的东西，不要整个拽进来。

有不清楚的？问我。这一课常见的卡点： 「防腐层和第 7 课的仓储到底啥区别？」（——形状一样，挡的东西不同：仓储挡数据库/技术，防腐层挡另一个上下文的模型） 「防腐层是不是就是加个 DTO 转换？」（——DTO 转换是它的手段之一，但重点在意图：主动隔离，不让上游的概念/词汇渗进你的领域） 「上下文映射是要画 UML 吗？」（——不必。它先是一种认识：项目里有几个上下文、每条连线什么关系。一张便签就能起步）
