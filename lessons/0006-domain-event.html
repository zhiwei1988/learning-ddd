领域事件：边界外面的事怎么办

上一课你学会了画边界，也学会了一句硬规矩：一个事务只改一个聚合。 可现实里，订单一确认，库存要扣、积分要加、短信要发——它们全在别的聚合里。这一课就补上那个出口。

你在地图上的位置：第 5 课你把「下单即扣库存」判在了聚合边界外， 理由是它不是不变量、是业务流程。判得对——但当时留了个洞：边界外的事，到底怎么发生？ 这一课给答案。随时打开 DDD 全景地图 定位。

先热身：回顾检查

两道回扣前两课的题。凭记忆作答。

问题：聚合是座孤岛

你的生鲜电商，一条业务事实：「订单确认了。」紧跟着一串后果—— 扣库存、加积分、发确认短信、通知仓库拣货。

可你上一课刚立下规矩：Order 只管自己那个圈，一个事务只改一个聚合。 于是 Order.confirm() 里不能去动库存聚合，更不能调短信服务——那会把四个聚合锁进一个大事务， 正是第 5 课警告过的「大圈、大锁、烂并发」。

那这些后果谁来触发？最糟的答案是让 OrderService 在确认订单后， 手动一个个去调 inventory.deduct()、points.add()、sms.send()—— Order 于是隐式知道了全世界，每加一个后果就得回去改它。耦合又长回来了。

解法：把「已发生的事」变成一等公民

Evans 的处方，一句话定义：

Something happened that domain experts care about. （一件领域专家关心的事，发生了。） — Eric Evans, DDD Reference，"Domain Events"（第 13 页）
Therefore: Model information about activity in the domain as a series of discrete events. Represent each event as a domain object. （因此：把领域中的活动建模成一串离散的「事件」，把每个事件表示成一个领域对象。） — DDD Reference，"Domain Events"（第 13 页）

换句话说：Order 不去命令任何人，它只大喊一声事实—— OrderConfirmed！——然后就不管了。谁关心这件事，谁自己去订阅。 库存、积分、短信各自听到、各自反应。Order 根本不知道它们的存在。

一个字之差，是整个模式的分水岭： OrderConfirmed（订单已确认）是事件——过去式，陈述一个既成事实； ConfirmOrder（确认订单）是命令——祈使句，要求别人做事。 事件不是「请你扣库存」，而是「我这边确认了，与你无关」。发布者不假设任何人会做任何反应——这正是解耦的来源。

事件长什么样：三条，全是你已经会的

  1. 过去式、不可变的事实。它是历史，改不了。Evans： Domain events are ordinarily immutable, as they are a record of something in the past. ——不可变？没标识？按值相等？这不就是值对象吗。 你第 4 课就会写了，只是现在它有了名字。
  2. 携带 id 和快照，不携带引用。Evans 说事件里装 the identity of entities involved in the event——是标识，不是指针。 这是第 5 课「跨聚合只用 id」的同一条纪律：事件里放 CustomerId、放当时的 total 快照， 绝不放 Order*。放了指针，订阅者就能顺着它反向篡改聚合——和上一课那个泄漏 getter、 第 4 课那个 Product& 是同一种病。
  3. 由聚合根产生，和状态改变同时发生。根是唯一的门（第 5 课）—— 状态变成「已确认」的那一刻，正是事实诞生的那一刻，事件就在这里被记下。不早不晚。

C++：事件是值对象，事件集是 std::variant

先把事件写出来。注意它和 Order 的刺眼对比——同一个 = default， 在这里是对的，在 Order 上是头号事故（这正是你回顾检查第 2 题在考的）：

// A domain event: past-tense fact, immutable, equal BY VALUE.
struct OrderConfirmed {
  OrderId    order;
  CustomerId customer;   // identity of another aggregate — held BY ID
  Money      total;      // a snapshot of the fact, NOT a live handle
  TimePoint  at;
  bool operator==(const OrderConfirmed&) const = default;  // ✓ correct: it's a value
};

// The CLOSED set of facts this aggregate can emit. A sum type — the C++ way to
// say "one of these, and the compiler knows every one." std::visit then forces
// you to handle each; add a new event and the compiler points at what's missing.
using OrderEvent = std::variant<OrderConfirmed, OrderCancelled>;

这是 Java 教材给不了的角度。那边领域事件通常是一棵接口继承树——开放的， 编译器不知道一共有几种。C++ 的 std::variant 把「这个聚合只会发这几种事」 变成一个封闭的类型，std::visit 逼你把每一种都处理掉。少写一个 case，编译期就红了。

再看根怎么产生事件——它只是记下来，不亲自去执行任何后果：

class Order {                      // the aggregate ROOT
 public:
  // The fact and the state change happen together, in ONE operation.
  void confirm(TimePoint now) {
    if (confirmed_) throw std::domain_error{"order already confirmed"};
    confirmed_ = true;
    events_.push_back(OrderConfirmed{id_, customer_, total_, now});
  }

  // The application layer pulls the facts and publishes them AFTER this
  // aggregate's transaction commits. Pulling clears the buffer: released once.
  std::vector<OrderEvent> pull_events() { return std::exchange(events_, {}); }

 private:
  CustomerId              customer_;   // held BY ID (lesson 5)
  bool                    confirmed_ = false;
  std::vector<OrderEvent> events_;     // facts recorded, not yet announced
};

关键在时序——这就是第 5 课那句「跨边界异步」落到代码上的样子：

order.confirm(now);              // 1. change ONE aggregate, commit its transaction
saveToDb(order);                 //    (order is now durably "confirmed")

for (auto& e : order.pull_events())
  publish(e);                    // 2. AFTER the commit, announce — subscribers
                                 //    (inventory, points, sms) react in their own
                                 //    transactions, eventually consistent
Within an aggregate boundary, apply consistency rules synchronously. Across boundaries, handle updates asynchronously. （边界内同步保证一致；跨边界异步更新。） — Eric Evans, DDD Reference，"Aggregates"（第 16 页）

护栏：别把什么都做成事件

和「别把聚合画大」是同一个警惕心。你已经知道 DDD 不是处处适用， 这里是它的又一个战术版：只有领域专家真正关心的事实，才配当领域事件。Evans 说得直白：

Ignore irrelevant domain activity while making explicit the events that the domain experts want to track or be notified of. （忽略无关紧要的领域活动，只把领域专家想追踪、想被通知的那些事显式建模出来。） — DDD Reference，"Domain Events"（第 13 页）

「用户点了一下按钮」是 UI 事件，不是领域事件；「某字段被 set 了一次」是实现细节，不是领域事件。 给每个 setter 都发一个事件，你会淹死在噪音里，业务方一个都不关心。判据只有一个： 业务方会说「哦，这件事很重要，我要知道它发生了」吗？

本课首选延伸阅读：继续读 Vaughn Vernon 的 Effective Aggregate Design 的 Part III（免费，dddcommunity.org 官方）。上一课你读了 Part I（聚合要小）； Part III 正好接上这一课——他演示如何用领域事件让两个聚合达成最终一致：一个聚合在自己的事务里改完、 发出事件，另一个聚合在稍后的事务里响应。你这一课的 confirm → pull → publish 就是它的骨架。 一手定义见 DDD Reference 第 13 页，本课引文都出自那一页 （注意标题上的 *——领域事件是 Evans 在蓝皮书之后才补进模式语言的三个模式之一）。
