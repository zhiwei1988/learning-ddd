限界上下文：一个词，两个意思

上一课要你把语言统一。这一课告诉你统一到哪儿为止——以及为什么「全公司统一」是个陷阱。

你在地图上的位置：还在战略设计。第 1 课学的 统一语言 有个前提没说破：一门语言只在某个边界内才是精确的。 这一课就是那个边界。随时打开 DDD 全景地图 定位。

上一课藏了半句话

Evans 给统一语言的官方定义，你上一课只读到了前半截。完整版是这样的：

A language structured around the domain model and used by all team members within a bounded context to connect all the activities of the team with the software. — Eric Evans, DDD Reference，"Definitions"（工作区内附官方 CC BY 4.0 副本）

within a bounded context——这五个词是整个 DDD 战略设计的枢纽。统一语言从来不是「全公司说同一套词」， 而是「在一个边界内说同一套词」。跨过边界，同一个词换个意思，是合法的，甚至是必须的。

为什么不能全公司统一？因为有人认真试过，然后失败了：

Total unification of the domain model for a large system will not be feasible or cost-effective. （对一个大型系统的领域模型做完全统一，既不可行，也不划算。） — Eric Evans，转引自 Martin Fowler，Bounded Context

Fowler 举过一个电力公司的例子：「电表（meter）」这个词在公司内部有三种互不相同的含义—— 电网与某个地点的接入点、电网与某个客户的接入关系、以及墙上那个物理装置。 这三拨人开会时用同一个词毫无障碍，因为人脑会自动按语境消歧。 但软件不会。软件要求的精确度，容不下这种模糊。

关键转折：模糊不是因为大家没沟通好，而是因为业务本身就分化了。 销售的世界和仓库的世界确实不一样。想靠开会把它们统一成一个词，是在跟现实作对。 DDD 的做法反过来：承认分化，然后给每个分化的世界画一条明确的线。

不画线的下场：那个谁都不敢删字段的 Order

每个公司都有这么一个类。它一开始很干净，然后每个团队往里加自己的字段，谁也不敢删别人的。

右边有三处值得停下来看：

  • 物流那边根本不叫 Order，叫 Shipment。边界给了你重命名的自由——在仓库主管的语言里，那就是「一个要送到某地址的箱子」，不是「订单」。名字跟着语言走，而语言跟着上下文走。
  • 跨界只留一个 id（SalesOrderId），不是共享对象。边界之所以成立，靠的就是不共享模型。共享了类型，边界就只是画在纸上。
  • 那个万恶的 status 消失了。因为它本来就是三个概念被硬塞进一个 int：销售关心「付没付」，物流关心「发没发」，计费关心「开没开票」。拆开之后，每个上下文的状态都能用自己的语言精确表达。

销售上下文 Sales

语言的主人：销售、运营

  • 订单 = 客户承诺购买的东西
  • 关心：优惠券、客单价、是否已支付
  • 「已完成」= 钱到账了

履约上下文 Fulfilment

语言的主人：仓库、物流

  • 包裹 = 要送到某地址的箱子
  • 关心：重量、承运商、拆包补发
  • 「已完成」= 客户签收了

注意「已完成」——同一个词，两个上下文里是两个完全不同的事实。 如果你把它们塞进一个 status 字段，你就得永远回答「已完成到底是谁的已完成」这个问题。

边界画在哪儿？Evans 说得很具体

限界上下文的官方定义只有一句话：

A description of a boundary (typically a subsystem, or the work of a particular team) within which a particular model is defined and applicable. （一条边界的描述——通常是一个子系统，或某个特定团队的工作范围——在这条边界内，某个特定的模型有定义、且适用。） — Eric Evans, DDD Reference，"Definitions"

而模式条目里的行动指令（Evans 的原文用 Therefore: 引出）告诉你这条线要划在哪些地方——注意，它不只是代码里的一条线：

  • 团队组织（team organization）——谁在维护这个模型
  • 应用的特定部分（usage within specific parts of the application）
  • 物理产物：代码库与数据库 schema（code bases and database schemas）

换句话说，一条真边界是有牙齿的：不同的团队、不同的代码库、不同的表。 如果两个「上下文」共享同一套表、同一批类，那它们只是文档里的两个词，不是两个限界上下文。 这是最常见的假 DDD——包名分了，模型没分。

你现在获得的技能：用语言探测边界。不要一上来就问「系统该怎么拆」。 去听业务方说话，找两种信号：
① 同词异义——同一个词，两拨人说的不是一回事（电表、订单、客户）。
② 异词同义——两拨人用不同的词说同一件事（销售叫「订单」，仓库叫「出库单」）。
这两种信号出现的地方，就有一条边界在等着被画出来。
