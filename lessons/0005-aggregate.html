聚合：一次改动该覆盖多大范围

战术设计最难、也最值钱的一块。前一课你学会了怎么给对象分类；这一课你要学会在它们之间画一条线——线内同时成立，线外各管各的。

你在地图上的位置：你已经能把一个概念判成实体或值对象了。 但真实系统里对象从不单独存在——它们互相牵扯。聚合就是回答：这堆互相牵扯的东西，哪些必须捆在一起改？ 随时打开 DDD 全景地图 定位。

先热身：回顾检查

两道回扣第 4 课的题。凭记忆作答。

问题：规则不长在单个对象身上

先看一条业务规则，来自你的生鲜电商：

「一张订单的总额，不得超过该客户的信用额度。」

请你认真想三秒：这条规则该由谁来保证？

OrderLine 不行——它只知道自己那 60 块，不知道兄弟们加起来多少。 Money 更不行，它连自己属于哪张订单都不知道。 这条规则不长在任何一个对象身上，它长在一群对象的「总和」上。 于是它无家可归，最后往往漂到 service 层、controller 层，甚至前端——领域模型退化成一堆数据袋子。

更糟的是并发。两个人同时往这张订单里加东西，各自加之前都检查了「没超额」，各自都通过， 最后总额超了两倍——而每一步都是「合法」的。Evans 开篇讲的就是这个：

Objects are supposed to maintain their own internal consistent state, but they can be blindsided by changes in other objects that are conceptually constituent parts. （对象本该维护自己内部状态的一致性，但它们会被「概念上属于自己组成部分」的其他对象的变化打个措手不及。） — Eric Evans, DDD Reference，"Aggregates"（第 16 页）

解法：画一条线，指定一个守门人

Evans 的处方，这段值得逐句读——整个战术设计里信息密度最高的一段：

Therefore: Cluster the entities and value objects into aggregates and define boundaries around each. Choose one entity to be the root of each aggregate, and allow external objects to hold references to the root only […]. Define properties and invariants for the aggregate as a whole and give enforcement responsibility to the root […]. Use the same aggregate boundaries to govern transactions and distribution. （因此：把实体和值对象聚成「聚合」，为每个聚合划定边界。为每个聚合选一个实体做「根」， 只允许外部对象持有对根的引用。为聚合整体定义属性与不变量，并把「强制执行」的责任交给根。 用同一条聚合边界来管辖事务与分布。） — Eric Evans, DDD Reference，"Aggregates"（第 16 页）

拆成四件事，你只要记住它们：

  1. 边界——一圈线，圈住一簇实体和值对象。
  2. 根——圈里选一个实体当门。外面的人只能跟它说话。
  3. 不变量——那条「总和不得超额」的规则，属于整个圈，由根来守。
  4. 事务——这条线同时就是事务的线。圈内立刻一致，跨圈事后一致。

第 4 点是最多人漏掉、也最有杀伤力的一句。Evans 紧接着补了刀： Within an aggregate boundary, apply consistency rules synchronously. Across boundaries, handle updates asynchronously. （边界内同步保证一致；跨边界异步更新。） 换句话说：一个事务只改一个聚合。你画聚合的时候，其实是在画事务。

边界画在哪：让不变量说话

新手常问「订单要不要把客户装进来？」——问法就错了。正确的问法是：哪些东西必须同时正确？

Order 聚合

不变量：总额 ≤ 信用额度

  • Order ← 根（唯一的门）
  • OrderLine ×N（内部实体）
  • ShippingAddress（值对象）
  • CustomerId ← 只是一个 id

Customer 聚合

不变量：它自己的那些规则

  • Customer ← 根
  • 信用额度、等级、联系方式
  • 它不知道订单的存在
  • 改它，跟改订单是两个事务

为什么客户不在订单圈里？因为没有任何规则要求「客户和订单必须同时正确」。 客户改个手机号，跟这张订单的总额毫无关系——硬捆在一起，你就是在为一条不存在的规则付事务的钱。 于是订单只持有 CustomerId，这是 Vaughn Vernon 那条著名的 「跨聚合只用 id 引用」（Effective Aggregate Design）。

这条规则对你有额外的红利：C++ 里「按 id 引用」意味着 Order 根本不需要 Customer 的定义—— 不用包含它的头文件，不用管它的生命周期，没有悬垂指针，编译依赖也断了。 你为了编译速度做了十年的解耦，DDD 说那正好是正确的领域建模。

C++：根就是那道门

把上面四件事翻译成代码。注意成员的持有方式——那才是边界的真身：

class Order {                       // ← the aggregate ROOT
 public:
  Order(OrderId id, CustomerId customer, Money credit_limit);

  // The ONLY way in. The invariant is checked here, on the whole
  // aggregate, before any state changes.
  void add_line(Sku sku, int qty, Money unit_price) {
    OrderLine candidate{std::move(sku), qty, std::move(unit_price)};
    const Money would_be = total().plus(candidate.subtotal());
    if (credit_limit_ < would_be)
      throw std::domain_error{"order total would exceed credit limit"};
    lines_.push_back(std::move(candidate));   // only now do we mutate
  }

  Money total() const;

  // Read-only window. NOT a mutable handle. See the trap below.
  const std::vector<OrderLine>& lines() const { return lines_; }

 private:
  OrderId                id_;
  CustomerId             customer_id_;  // another aggregate, held BY ID
  Money                  credit_limit_;
  std::vector<OrderLine> lines_;        // parts, owned BY VALUE
};

读这段代码的正确姿势——三行 private 成员就把整个模式说完了：

  • std::vector<OrderLine> lines_ —— 按值持有。订单行是订单的一部分， 随订单生死，外面没人能单独拿到一个 OrderLine。不是 vector<shared_ptr<OrderLine>>—— 那等于公开宣布「欢迎外面的人抓着我的零件」，边界当场作废。
  • CustomerId customer_id_ —— 按 id 持有。客户是另一个聚合，隔壁老王，不归我管。 不是 Customer*，也不是 shared_ptr<Customer>。

所以在 C++ 里，聚合边界不是注释，是类型。值 = 圈内，id = 圈外。 读一眼成员声明就知道这条线画在哪——这是 Java 里享受不到的清晰度（那边一切皆引用，两种情况长得一模一样）。

护栏：别把聚合画大

你已经知道「DDD 不是处处适用」，这里是同一个警惕心的战术版。 新手学完聚合的第一反应是画个大圈——「订单、客户、库存、物流都相关，都圈进来，多安全」。 这是灾难。圈越大，事务越大，锁越大，并发越差。Evans 第一句就点了名： Cautious database locking schemes cause multiple users to interfere pointlessly with each other and can make a system unusable. （谨小慎微的数据库锁策略会让用户之间毫无意义地互相干扰，甚至让系统没法用。）

「毫无意义地」（pointlessly）是关键词。只有真正的不变量才配得到一条边界。 「下单后要扣库存」听起来很像必须同时发生，但它不是不变量，它是业务流程—— 电商界的现实是：先下单，库存事后扣，扣不出来就道歉退款。世界照转。 硬要它们强一致，你换来的是一个卡死的系统，和一条业务方从没要求过的保证。

一句话记住这一课：聚合边界 = 一致性边界 = 事务边界。 画线之前先问「哪条规则要求它们同时正确」——问不出规则，就别画那条线。

练一练：什么在圈里，什么在圈外

场景：生鲜电商的「销售上下文」，你正在设计 Order 聚合， 不变量是「订单总额不得超过客户信用额度」。每一项：在边界内，还是边界外？

主推读物

本课首选延伸阅读：Vaughn Vernon 的 Effective Aggregate Design（三篇 PDF，免费，dddcommunity.org 官方发布）。 这是聚合这个主题上公认最好的一手材料，没有之一——他从一个画得过大的聚合出发， 一步步重构到正确的边界，并提炼出四条经验法则（真不变量才建边界／聚合要小／跨聚合用 id 引用／边界外用最终一致性）。 先读 Part I（约 15 页），这一课基本就是它的前半段。示例是 Java，但你现在有能力自己翻译了。 一手定义见 DDD Reference 第 16 页 "Aggregates"，本课引文都出自那一页。
